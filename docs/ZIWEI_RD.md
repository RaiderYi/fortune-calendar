# 紫微斗数技术预研（Phase 6）

## 结论摘要

- **紫微斗数**排盘依赖完整的农历/节气、安星诀、四化与大量星曜规则表，工程体量显著高于当前以**八字**为核心的 `api/core/lunar.py` + `bazi_engine` 链路。
- 建议**不要**在短期迭代中与八字日运并行自研全量引擎，除非有专职命理工程与长期测试投入。

## 可选路线

1. **第三方库 / 商业 API**  
   - 评估维护成本、授权、数据出境与 SLA。  
   - 适合快速验证「紫微」入口与 AI 解读闭环。

2. **开源实现（若协议允许）**  
   - 需核对许可证、准确度与社区维护状态。  
   - 建议先做**固定命盘**黄金用例回归，再接入生产。

3. **自研子集**  
   - 仅实现命宫、身宫、主星少数颗 + 简单四化，可作为「轻量紫微」MVP，但需产品明确标注**非专业级排盘**。

## 与现架构的集成点

- 入口：与 `/app/fortune/*` 并列，例如 `/app/fortune/ziwei`。  
- 档案：复用出生时刻、性别、经度；注意**农历/节气**与八字立春换年规则差异，需在 UI 与文案中说明。  
- AI：`AIService` 可增加 `build_ziwei_system_prompt`，输入为结构化命盘 JSON（星曜、宫位、四化）。

## 风险

- 流派众多（中州、飞星、三合），易引发用户预期落差。  
- 合规：与八字模块相同，需强调娱乐参考、禁止医疗/投资断言。

## 建议决策

| 阶段 | 动作 |
|------|------|
| 当前 | 不阻塞八字、月运、问卦、合盘等主线。 |
| 预研 2 周内 | 调研 1～2 个可授权数据源 + 10 组命盘对照。 |
| 决策点 | 若 API 成本可控且准确度可接受 → 接 API；否则延后或做「轻量子集」演示。 |
